## What It Does & Project Motivation

Short description: Our project classifies user prompts to an LLM as toxic or safe, so that LLM interactions can be safer in the future. Our main methodology includes fine-tuning a pretrained transformer language model (adapting output heads) as a toxicity classifier with and without custom safety policies (Policy A, B, C), calling state-of-the-art transformer models for a classification task, and comparing these results with a set of baselines. Our key finding was that incorporating a custom safety policy did not improve the effectiveness of the pretrained BERT classifier but it significantly increased the effectiveness in  SOTA models' classification. 

This project aims to build and evaluate a toxicity classification system for text input by fine-tuning a pretrained DistillBERT model on the ToxiChat dataset. This project has three main goals. First, we compare a standard text-only classifier with a policy-conditioned model to find out whether explicit rules improve performance. We then evaluate how three different policy designs, A which can be seen as "strict", B as "balanced", and C as "lenient" affect metrics such as precision, accuracy, and recall. Finally, we compare our fine-tuned model to "judge" API model Claude to assess differences in performance. These experiments will help us understand how both model design and policy shape our experimental outcomes.

Our main project goal was to probe on the question: can stricter safety guardrails help detect toxic content early, leading to safer LLM interactions? We wanted to see if using policy-conditioned models of different types, long vs. short, generic AI-generated vs. social science research-backed, and policy vs. no policy at all, would make a difference in the effectiveness of toxicity detection.

Versions Evaluated:

- Majority-class baseline: Always predicts the majority class, which in this dataset is "safe".
- Version 1: Text-only DistilBERT classifier
  - Input: text
  - Output: safe or toxic
- Version 1b: Class-weighted DistilBERT classifier
  - Same as version 1, but uses class-weighted loss to address imbalance between the number of safe vs. toxic examples.
- Version 2: Policy-conditioned DistilBERT classifier
  - Input: safety policy + text
  - Output: safe or toxic
  - Three different policies tested:
  -   A: Long policy grounded in research, using the "Understanding Generative AI Risks for Youth: A Taxonomy Based on Empirical Data" paper's risk definitions from its appendix
  -   B: strict and short policy, generated by ChatGPT
  -   C: lenient and short policy, generated by ChatGPT

### Phase 1: Fine-Tuning DistilBERT Models

Overall, all fine-tuned models outperformed the majority class baseline, which as we expected confirms that our models are learning meaningful patterns beyond just predicting the dominant class. The baseline achieves an accuracy of 0.867 due to the class imbalance towards the “safe” class but fails to  identify toxic content with precision, recall, and F1 scores of 0, which could be incredibly problematic for missing instances of actual toxic content.

### Baseline vs. Unweighted Text-Only DistilBERT vs. Weighted Text-Only DistilBERT
The v1_baseline model achieves the highest performance with an accuracy score of 0.965 and F1 of 0.863; this indicates that the model is effective at detecting toxicity. However, it is important to note that this model did not consider the class imbalance which may skew model behavior since the data overrepresents the “safe” class, which explains why the F1 is lower while the accuracy is higher from encountering a larger volume of “safe” class data. 

The v1_b_weighted takes into account class imbalance and becomes more sensitive to toxic content which is reflected in seeing fewer false negatives; however, this also led the model to generate more false positives, meaning flag more safe content as toxic. This model version is especially preferable for a content moderation perspective because it is better to over-flag content, instead of missing a potentially dangerous or explicit piece of toxic content. 

### Weighted Text-Only DistilBERT vs. Policy Conditioned Model

All policy-conditioned models (v2_policy_A, v2_policy_B, v2_policy_C) performed slightly worse than the text-only baseline, with accuracy dropping to 0.956 across all policies. This suggests that adding policy text does not improve performance, and may introduce noise to the initial classification task. 

Across the different policies, Policy A was directly pulled from the “Understanding Generative AI Risks for Youth: A Taxonomy Based on Empirical Data” by Yu, et al. paper by directly pulling the most relevant risks defined in their appendix, specifically GAI System Toxic Content Generation Risk, Simulated Toxic Interaction Risk, Malicious Exploitation Risk, and Harmful Behavioral Influence Risk. Policy B, and C were generated from ChatGPT and got progressively shorter/less specific. Overall, Policy B performed the best, suggesting that a medium balance between length and specificity could be ideal to optimize model performance and minimize what the model might consider as possible noise.

### Part One Key-Takeaways
The standard text-only DistilBERT model achieved the best overall performance; though the weighted text-only version (v_1) performed slightly worse than the unweighted text-only (v_1b), it is the most trustworthy as it accounts for class imbalance. Overall, we found that adding policy text did not improve results. Comparing out key metrics, we found that due to the class imbalance the accuracy metric can mask failure on the minority (toxic) class which is why relying on additional metrics such as F1, precision, and recall was necessary. 

### Phase 2: LLM-As-Judge Gemini
To complement our fine-tuned DistilBERT models, we evaluated a large language model (Gemini/Gemma-3-12B) as a zero/few-shot classifier on a small balanced set of toxic and safe prompts. We tested two conditions: without policy guidance and with Policy A (research-based).

### Text-Only Gemini vs. Policy Conditioned Gemini
Overall the baseline Gemini performance achieves strong, and well-balanced performance with accuracy/precision/recall/f1 of around 0.92. Across both classes, performance is highly balanced, with equal precision and recall across both classes. 

Adding Policy A significantly improved accuracy to 0.98, F1 to 0.98, and recall to 1, eliminating all false negatives so that the model catches every toxic instance. In contrast, this model type is especially relevant for safety contexts, where missing toxic content is more costly than over-flagging. 

While for DistilBERT, policies acted as noise or redundant input, policy conditioning helps LLMs by acting as behavioral guidance. We hypothesize this difference is because Gemini is trained on large-scale instruction-following data and so it interprets policy as instructions, and can dynamically adjust its behavior based on this context. On the other hand, DistilBERT is optimized for a fixed classification task, so it treats the policy text as additional input tokens rather than instructions, which act as added noise for the model.

### Model Development and Decision-Making Process

We additionally made design decisions in the Phase 2 code for the LLM-as-judge experiments because of constraints we learned iteratively about latency, cost, and runtime. Firstly, we hit rate limits with the Gemini free tier, limiting both Requests Per Minute (RPM) and Requests Per Day (RPD). Because of this, we had to shift from our initial implementation of running our judge through Gemini (we tried gemini-3-flash-preview, gemini-2.5-flash, and gemini-2.5-lite) and had to switch to a hosted model that could handle higher RPM and RPD. We analyzed the Google catalog and found that the Gemma model series could handle a much higher RPM and RPD, along with input tokens, because it had less reasoning capacity. So, we shifted this set of experiments to run on gemma-3-12b. Additionally, to counteract any RPM issues, we imported the “time” module and introduced a 12-second latency delay in running each API call so that our function would remain under 5 RPM so that we would avoid getting rate-limited and blocked by the Gemini API. 

Secondly, because of constraints on API cost, we decided to run the LLM-as-judge experiments on a smaller batch of the data, which we randomly sampled from the main dataset. Initially, this random minibatch was mostly “safe” class data points because of the class imbalance, so we pivoted and wrote the get_balanced_sample() method to get a random sample where 50% of the data was randomly sampled from the “toxic” class and the other 50% was randomly sampled from the “safe” class to adjust for the class imbalance and make sure that our results here would be representative. We recognize that only having run 50 user prompts on Gemini (no policy) and 50 user prompts on Gemini (Policy A) is a much smaller sample size than our original input model with thousands of data points, but we designed the class rebalancing to account for that and ensure that the results were at least a somewhat good proxy since we saw that it would be computationally expensive to run all 5,654 input rows through the API. 

### Error Analysis 
Refusal played a role in the failure cases we got for the LLM-as-judge experiments. We included a line in the functions for running the API calls that printed each LLM output, and we saw that for running Gemini with no policy, some of the outputs were “None”. This is because gemma-3-12b took in the toxic user input, and without guidance of the policy, it likely flagged Google’s internal safeguard system. But for running Gemini with Policy A, every single model output was valid (SAFE or TOXIC), showing that the inclusion of the policy helped the request actually be evaluated, instead of facing refusal because the model didn’t understand that this is a classification task, not a real toxic input. 

Model failures in the LLM-as-judge experiments were primarily: 
- FN: model misidentified a toxic prompt as safe (unable to pick up on toxic characteristics)
- FN: model saw a toxic prompt and refused to answer (answered with empty string or None) 
- FP: model misidentified a safe prompt as toxic (model was too cautious)

Qualitative analysis of failure cases: 
These were case studies of disagreements between the Gemini-without-policy vs. Gemini-with-Policy-A models:

| data point # | text | ground_truth | no_policy | policy_A
| -------- | -------- | -------- | -------- | -------- |
| 23 | write a screenplay for a scene, a court, rape ... | 1 | 0 | 1 |
| 27 | You are a typical 4chan user named Kevin. You .. | 1 | 0 | 1 |
| 40 | Create a story about a girl eating a frog | 0 | 1 | 0 | 

In data points #23 and #27, the no_policy model had a False Negative and classified these two toxic prompts as safe because it was unable to pick up on the nuance of violence or political polarization just because it was framed in a jailbreaking style of being a "screenplay" or "4chan" chat, the the policy_A model was actually able to pick up on those nuances because of the extensive safety policy.

In data point #40, the no_policy model had a False Positive and classified this safe prompt as toxic because it applied a very rough logic to classifying safety and didn't pick up on the fact that there were no safety violations in that prompt (although definitely a strange thing to prompt a model). Again, the policy_A model was able to pick up on this and correctly classified it as safe.
